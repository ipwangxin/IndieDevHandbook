# 怎么做一个足够小的 MVP

> *字数：约 1680 字；预计阅读时间：4 分钟。*


MVP 不是残缺产品，而是用最小成本验证一个关键假设。

独立开发最容易犯的错，是把第一个版本做成“完整平台”：登录、权限、支付、后台、通知、团队协作、数据看板一个都不想少。结果几周过去了，产品越来越完整，但最关键的问题还没回答：

> 真的有人需要它吗？

AI 和 vibe coding 会让这个问题更隐蔽。以前你做不完，是因为时间不够；现在 AI 能帮你很快堆出很多功能，于是你更容易误以为“既然做得快，那就都加上”。

但 MVP 的重点不是少写代码，而是少验证假设。

## MVP 要验证什么

一个好的 MVP 不应该试图证明“我能做很多功能”，而应该证明一件事：

> 用户愿意为了这个结果付出行动成本。

行动成本可以很轻，比如：

- 留下邮箱。
- 加群。
- 上传真实文件。
- 给你发样本。
- 愿意试用。
- 愿意付费。
- 愿意把它推荐给同事或朋友。

如果用户只是点赞、收藏、说“这个想法不错”，还不够。那是兴趣信号，不是需求验证。

## 缩到一个核心路径

一个好的早期 MVP 通常只保留一条核心路径：

1. 用户输入一个真实问题。
2. 产品给出一个比现有方法更省事的结果。
3. 用户能立刻判断这个结果有没有价值。
4. 你能收集下一步反馈。

例如“供应商报价 PDF 对比器”的核心路径可以是：

```text
上传 3 份报价文件 -> 生成一张对比表 -> 用户修正错误 -> 导出 Excel
```

“反菜谱做饭规划器”的核心路径可以是：

```text
输入已有食材和预算 -> 生成 3 顿饭 -> 合并购物清单 -> 用户替换不想吃的菜
```

这就够了。第一版不需要社区、不需要复杂账号、不需要移动 App，也不需要漂亮到像成熟产品。

## 暂时不要做的东西

早期最容易拖慢项目的，往往不是核心功能，而是“看起来迟早要做”的东西。

第一版可以先不做：

- 完整账号系统
- 复杂权限
- 团队空间
- 管理后台
- 多语言
- 发票
- 完整订阅系统
- 复杂数据看板
- 大量第三方集成
- 自动化到没有人工兜底

不是说这些永远不做，而是等用户证明“核心结果有价值”之后再做。

如果 AI 已经帮你生成了这些功能，也不一定要上线。删掉不必要功能，比保留一个没人用但会增加维护成本的功能更健康。

## AI 时代的 MVP

AI 更适合帮你做“能验证的第一版”，而不是帮你跳过验证。

可以让 AI 帮你：

- 写产品定义。
- 生成落地页文案。
- 搭工具页面。
- 写表单处理逻辑。
- 生成测试数据。
- 整理用户反馈。

但不要让 AI 替你决定：

- 哪个用户最值得先服务。
- 是否应该付费。
- 哪个功能一定要做。
- 这个方向是否值得继续。

AI 可以加快产出，真实用户才能验证方向。

## MVP 的三种形态

### 1. 表单型 MVP

用户提交信息，你手动或半自动处理，然后返回结果。

适合验证：

- 用户愿不愿意提交真实信息。
- 结果对用户有没有价值。
- 用户会不会继续追问或要求下一版。

例如：用户上传报价文件，你先人工整理成对比表。等需求成立，再自动化。

### 2. 工具型 MVP

用户在网页里完成一次明确操作，并立刻拿到结果。

适合验证：

- 工具是否能减少用户当前的麻烦。
- 用户是否愿意重复使用。
- 用户是否愿意为导出、批量、保存、提醒等能力付费。

例如：粘贴一段推广文案，生成适合知乎发布的改写版本。

### 3. 内容型 MVP

先用文章、案例库、清单或免费书聚集目标用户，再观察他们最关心的问题。

适合验证：

- 目标人群是否存在。
- 哪些问题最容易引发评论和私信。
- 后续应该做产品、社群、咨询、模板还是工具。

例如：这本《国内独立开发冷启动手册》本身就是内容型 MVP。

## 做完 MVP 后看什么

不要只看访问量。访问量很容易误导你。

更应该看：

- 有多少人完成核心动作？
- 有多少人愿意留下联系方式？
- 有多少人提供真实样本？
- 有多少人问下一步怎么用？
- 有多少人愿意付费或预约？
- 有多少人第二次回来？

如果有人认真反馈“不准”“这个字段不对”“能不能支持 Excel”，这反而是好信号。说明他们真的把它当工具用，而不是随便看看。

## 第一版的原则

第一版要能被用户理解，也要能被你快速丢掉。

不要把早期产品写得太重。MVP 的价值不是沉没成本，而是帮你用最短时间知道该继续、缩小、转向还是停止。
